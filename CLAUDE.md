# Parivartan Sutra — Flutter App

This file documents what actually exists in this Flutter project right now, found by reading the code directly — not what was planned or intended. Git history: 3 commits as of the original audit (`first commit` → `first code with structured` → `feat: Implement authentication flow with login, signup, and password recovery views`), plus the fixes below.

## Project overview

- Flutter app, package name `parivartan_sutra`, `sdk: ^3.10.3`.
- Architecture: GetX (`get: ^4.6.6`) for state management, routing, and DI — `lib/app/` (App widget, routes, initial bindings), `lib/core/` (network, storage, theme, session, errors), `lib/features/<feature>/{controllers,views,models,bindings,services}`, `lib/shared/widgets/` (a few reusable widgets).
- Dependencies (`pubspec.yaml`): `get`, `http` (raw `package:http`, not Dio), `flutter_secure_storage`, `shared_preferences`, `google_fonts`. No Retrofit/Dio, no code-gen (no `build_runner`/`json_serializable`/`freezed`), no image-loading lib (no `cached_network_image`), no local DB (no Hive/Isar/sqflite), no push-notification/Firebase packages, no analytics/crash-reporting packages, no state-mgmt alternative (no Bloc/Riverpod — pure GetX).

## Fixed (2026-07-22) — real backend integration pass

Everything below was verified against `flutter analyze` (clean, one info-level style lint) and `flutter build web --debug` (compiles). Real device/backend end-to-end verification (login → children list) still pending a running `php spark serve` from the user.

- **`lib/core/network/api_endpoints.dart` and `env_config.dart` rewritten to match the real backend exactly** (copied from `../parivartan-sutra-new/CLAUDE.md`'s "Mobile API" section and `app/Config/Routes.php`, not guessed): base URL is now `http://127.0.0.1:8080/api/` (plain `api/` prefix, no subdomain, no `/v1` — matches `app/Config/App.php`'s `baseURL` + the `api` route group), with `login`, `logout`, `interview/*`, `strategy/*`, `parent/children`, `parent/dashboard`, `student/dashboard` endpoints. All the old speculative paths (`/auth/send-otp`, `/child`, `/test/parent/start/*`, `/dashboard`) are gone.
- **`ApiClient` now sends form-urlencoded bodies, not JSON.** Confirmed every Mobile API controller (`AuthController`, `ParentController`, `InterviewController`, `StrategyController`) reads request bodies via CodeIgniter's `getPost()`, which only populates from `application/x-www-form-urlencoded`/multipart — not a raw JSON body. `post`/`put`/`patch` now pass a `Map<String, String>` body directly to `package:http`, which sets the correct content-type itself; the previous `jsonEncode(body)` + forced `Content-Type: application/json` would not have worked against this backend at all.
- **`ApiClient._handleResponse` now decodes the response body before branching on status code**, so the backend's real `message` (e.g. "Invalid password", "Mobile number not registered", the 429 throttle message) reaches the UI instead of a generic fallback string.
- **Login flow rewritten end-to-end for the real single role-detecting `POST login`** (mobile + password, no OTP):
  - `AuthUserModel` now matches the real response shape `{token, token_type, expires_at, user: {id, role, profile_id, name, mobile, email}}` — the old nested `AuthTokenModel` and the doc-comment claiming `access_token`/`refresh_token`/`expires_in` fields are gone (that shape never matched anything the backend actually returns).
  - `AuthApiService` now has one `login(mobile, password, {device})` and `logout()` — `sendOtp`/`verifyOtp`/`studentLogin`/`refreshToken` are gone (no OTP flow and no refresh endpoint exist server-side).
  - `LoginController` rewritten around `mobileController`/`passwordController` (no more `emailController`/`otpController`/`LoginMode`).
  - **`ParentLoginView` is now actually wired to `LoginController`** — it was previously a disconnected `StatefulWidget` with a no-op `onPressed: () {}` "Log In" button. It's now a `GetView<LoginController>` with a real `Form`, mobile+password validation, loading state, and error display bound to the controller.
- **`SessionManager.saveSession` now stores `role` and `mobile`** (new `AppConstants.keyUserRole`/`keyUserMobile`), with `isParent()`/`getUserRole()`/`getUserMobile()` helpers — needed since the backend's single login endpoint returns a role the client has to branch on. `getRefreshToken()`/`keyRefreshToken` removed (no refresh flow exists). `logout()` now calls the real `POST logout` (revokes the token server-side) before clearing local session.
- **`SplashView` fixed to check session state before routing.** It previously read a raw `SharedPreferences` key (`'seen_onboarding'`) that didn't match `AppConstants.keyOnboardingDone`, and never checked login state at all — so a logged-in user was always bounced back to onboarding/login on cold start. It now checks `SessionManager.instance.isLoggedIn()` first (goes straight to `/home` if true), then `SharedPrefsService`/`AppConstants.keyOnboardingDone` for the onboarding-seen flag. `OnboardingThreeView`'s "Get Started" button was updated to write that same key/service instead of the old raw-prefs call, so the two sides now agree.
- **Duplicate `lib/features/auth/bindings/auth_binding.dart` deleted** — it was byte-for-byte identical to `login_binding.dart` and never referenced from `app_pages.dart`.
- **`/home` rebuilt as the first real feature screen: a parent's children list**, replacing the old generic template dashboard. `HomeController` now calls `GET parent/children` (was the placeholder `GET /dashboard` returning generic `{id, title, description, image_url}` items with no relation to this app's domain). New `ChildModel` (`lib/features/home/models/child_model.dart`, replacing the deleted `home_model.dart`) matches the real response row: `{id, name, class, dob, has_login}` (backend's `students` table minus `user_id`, plus the derived `has_login` bool). `HomeView` shows a card per child (name, class, a login-created indicator) with pull-to-refresh; if the logged-in user's role isn't `parent`, it shows an explicit "Student dashboard not built yet" placeholder rather than silently showing nothing or crashing.
- **Theme now matches the Parent web portal's "Sprout" design system** (`../parivartan-sutra-new/public/assets/css/parent-theme.css`) instead of an unrelated blue/green palette:
  - `AppColors`: `primary` → Sprout teal `#1FB8BE` (was `#004DA0` blue), `secondary` → Kite blue `#0B4DA2`, `accent` → Marigold `#F5A623` (was green `#22C55E`, no web equivalent), `error`/`errorLight` → Blossom `#E8637A`/`#FCE6EA` (Sprout has no separate red), text colors → Ink/Ink-soft `#4A4A4A`/`#7A7A7A`, surfaces/borders → Cloud/Border `#F5FAFB`/`#DCEBEC`. Field names were kept as-is (`primary`, `secondary`, `accent`, ...) so every existing usage picked up the new values automatically; only `primaryLight` deviates from a literal CSS var — it's a lightened *teal* (`#6FD3D8`) rather than the near-white `--ps-sprout-light` background tint, since the app uses `primaryLight` as a legible border/gradient/foreground color, not a background wash (that pale tint is kept separately as `AppColors.primarySurface`).
  - `AppTextStyles` now pulls **Baloo 2** (display/headline/title styles) and **Nunito Sans** (body/label/button/input styles) via the `google_fonts` package — no bundled font asset files needed. `AppTheme`'s base `fontFamily` also switched from `'Roboto'` to Nunito Sans.
  - Dark-mode colors were left as their original (non-Sprout) values — the web portal doesn't define a dark theme to match against, so there was nothing to align them to.

## Still open / not done in this pass

- **No real-device/real-backend verification yet.** `flutter analyze` and `flutter build web --debug` both pass, confirming the code compiles, but nobody has run this against a live `php spark serve` yet. `EnvConfig`'s dev `baseUrl` (`http://127.0.0.1:8080/api/`) works for the macOS/iOS simulator or Chrome; **an Android emulator needs `10.0.2.2` instead of `127.0.0.1`**, and a physical device needs the host machine's real LAN IP (or the ngrok URL commented out in the backend's `.env`).
- **Only the parent login → children-list path was rebuilt.** Student login still routes through the same `LoginController`/`AuthUserModel` (the backend's `login` is role-detecting either way), but `HomeView` only shows a placeholder for the `student` role — a real student dashboard (`GET student/dashboard`) isn't built.
- **Interview and strategy features still don't exist in the app** — `ApiEndpoints.interview*`/`strategy*` constants are defined (matching the real backend) but no controller/view calls them yet.
- **Unregistered routes remain unregistered, on purpose for now.** `AppRoutes` still declares `register`, `forgotPassword`, `resetPassword`, `verifyOtp`, `profile`, `editProfile`, `settings`, `notifications`, `changePassword`, `about` with no `GetPage` entry in `app_pages.dart`. `ForgotPasswordView`/`SignUpView` are still reached only via ad-hoc `Get.to()` calls from `ParentLoginView`, not the routes table. **`profile`, `editProfile`, `settings`, `notifications`, `changePassword`, `about` have no view files at all — not built yet.**
- `lib/test/widget_test.dart` is still the untouched default Flutter counter-app template test — doesn't test anything in this app, not touched in this pass.
- No `.env`/environment-file mechanism — `EnvConfig` still hardcodes URLs directly in source; prod `baseUrl` is a placeholder (`https://api.parivartansutra.com/api/`), not a confirmed real domain — nothing is deployed there yet.
- Sign-up, forgot-password, OTP-verification, reset-password views: still static UI only, not wired to any controller/API — out of scope for this pass (backend has no register/forgot-password/OTP endpoints to wire them to yet either).

## Backend Mobile API — reference summary (from `../parivartan-sutra-new/CLAUDE.md`)

Key points relevant to Flutter work, read from the backend's own documentation:

- Token-based (`Authorization: Bearer <token>`), lives entirely under `app/Modules/Api/`, parallel to the existing session-based web app (doesn't touch it).
- One role-detecting `POST login` (`mobile`, `password`, optional `device`) for both parent and student — no OTP flow exists server-side. Tokens are 30-day, long-lived, no refresh endpoint (re-login to get a new one).
- Every endpoint reads its body via CodeIgniter's `getPost()` → **requests must be `application/x-www-form-urlencoded`, not JSON** (now handled by `ApiClient`, see "Fixed" above).
- `apiauth` filter (optionally `apiauth:parent` / `apiauth:student`) protects routes; wrong-role token → 403, missing/bad/expired/revoked token → 401.
- Full endpoint list, exact paths, and verification notes: see `../parivartan-sutra-new/CLAUDE.md`, section "Mobile API (added 2026-07-22)".
- Response envelope `{"success": bool, "data": ..., "message": "..."}` — matches what `lib/core/network/api_response.dart` already expects.
